The LLM assists. The code decides.

This app uses built-in engineering logic as the source of truth. Claude can help interpret confusing answers and make responses easier to understand, but it does not calculate wind loads or choose ASCE values.

Deterministic app logic: validation, state, lookups, calculations
Optional LLM layer: interpretation, clarification, response polish
Safety gate: backend validation before anything is trusted

Step 1

User answers the current question

The frontend sends the message to the backend session route. The app already knows which question it asked.

Step 2

Deterministic controller parses first

Simple answers like numbers, yes or no, roof type, exposure category, and design standard are handled by code.

Optional help

If the answer is fuzzy, Claude may interpret

Claude can suggest a structured field update or ask a clearer follow-up. It is treated as a candidate, not a fact.

Safety gate

Backend validation checks the candidate

The same parsers and rules must accept the value. If validation fails, the app asks the user to clarify instead.

Step 3

Session state updates only after validation

The app stores structured inputs and advances to the next known question in the seven-phase flow.

Optional polish

Claude may rewrite the response for humans

It can make the wording friendlier, preserve the engineering content, and guide users toward better answers.

No arithmetic by the LLM Wind pressure math runs in the Python calculation engine.
No ASCE guessing Coefficients, lookups, and defaults come from deterministic code and data.
No flow control by the LLM The backend owns phase progression, branching, and confirmation.
Fallback always works If Claude is off or fails, the built-in question flow continues.
Hallucination prevention pattern Let the LLM improve communication, then let deterministic code decide what becomes engineering state.
